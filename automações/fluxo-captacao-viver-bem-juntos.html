Fluxo de captação de beneficiários com condições crônicas ativado via régua de captação no chatbot Blip, com o objetivo de engajá-los no modelo assistencial Viver Bem Juntos.

6.1 Autenticação — Azure AD (OAuth 2.0 Client Credentials)

A autenticação com o CRM Dynamics 365 usa o fluxo client_credentials do Azure Active Directory. O token obtido é enviado como Bearer em todas as chamadas subsequentes à Dataverse Web API.

OperaçãoMétodoEndpointJSON
Obter Token Azure AD POST https://login.windows.net/{tenantid}/oauth2/token
ℹ️ Extração automática do token O script de teste Postman (aba Tests) extrai automaticamente o access_token do retorno e salva na variável token da collection via pm.collectionVariables.set("token", jsonData.access_token).
6.2 Fluxo de Integração — Passo a Passo

A Plataforma Viver Bem é a origem do fluxo de automação de Alta Complexidade. Ela chama as APIs do CRM Dynamics 365 para criar/iniciar a Linha de Cuidado e, a partir dela, o próprio CRM aciona o BLIP via automação interna para disparo dos templates WhatsApp.

  1. Plataforma Viver Bem — Obter Token — Autentica no Azure AD (client_credentials) e obtém o Bearer Token do CRM.
  2. Plataforma Viver Bem — Verificar Contato por CPF — Consulta contacts filtrando pelo campo customizado uni_cpf.
  3. Plataforma Viver Bem — Criar / Atualizar Contato — Se não existir, cria via POST /api/data/v9.2/contacts. Se existir, atualiza telefone e e-mail via PATCH.
  4. Plataforma Viver Bem — Verificar Linha de Cuidado por CPF — Consulta uni_linhadecuidados expandindo o contato vinculado.
  5. Plataforma Viver Bem — Criar / Atualizar Linha de Cuidado — Persiste o vínculo do beneficiário no programa (estágio, elegibilidade) e inicia a automação de Alta.
  6. CRM → BLIP — A automação interna do Dynamics 365 detecta a criação/atualização da Linha de Cuidado e comunica o BLIP informando o template que deve ser disparado.
  7. BLIP → WhatsApp — BLIP envia o template ao beneficiário e registra o retorno em uni_whatsapps no CRM para auditoria.
ℹ️ Origem do fluxo Quem inicia a comunicação com o CRM é a Plataforma Viver Bem — não o BLIP. O BLIP atua como consumidor do CRM (via automação interna do Dynamics), responsável apenas pela mensageria WhatsApp.
6.3 Contato — Beneficiário no CRM contacts

Entidade padrão contacts do Dynamics 365 com campo customizado uni_cpf para identificação do beneficiário Unimed. Campos relevantes: fullname, telephone1 e emailaddress1.

OperaçãoMétodoEndpointJSON
Obter Por CPF GET /api/data/v9.2/contacts?$filter=uni_cpf eq '{cpf}'
Atualizar Contato PATCH /api/data/v9.2/contacts({contactid})
ℹ️ Criação de Contato A criação usa POST /api/data/v9.2/contacts (operação nativa do Dynamics 365, não mapeada explicitamente na collection). O campo uni_cpf deve ser preenchido na criação para viabilizar consultas subsequentes por CPF.
6.4 Linha de Cuidado uni_linhadecuidados

Entidade customizada do CRM que representa o vínculo do beneficiário ao programa de cuidado Viver Bem. Os flags de monitoramento alimentam as automações do Dynamics 365 que acionam o BLIP para disparo de mensagens WhatsApp.

OperaçãoMétodoEndpointJSON
Obter Por CPF GET /api/data/v9.2/uni_linhadecuidados?$expand=uni_Contact&$filter=(uni_Contact/uni_cpf eq '{cpf}')
Obter Por ID GET /api/data/v9.2/uni_linhadecuidados({id})
Criar Linha de Cuidado (Alta) POST /api/data/v9.2/uni_linhadecuidados
Atualizar Flags PATCH /api/data/v9.2/uni_linhadecuidados({id})
Deletar DELETE /api/data/v9.2/uni_linhadecuidados({id}) —
ℹ️ Body específico para o fluxo de Alta Complexidade A criação da Linha de Cuidado no fluxo de captação Viver Bem Juntos — Alta utiliza um payload diferente dos demais fluxos: sem os flags de monitoramento e incluindo uni_estagiodajornada (valor 122880000) e uni_elegivelaoprogramacronico: true.

7

Script de Captação — Alta Complexidade → CRM Viver Bem

Script Python que alimenta a fila de captação a partir da base de Vidas da Unimed POA, selecionando os 100 pacientes prioritários de Alta Complexidade sem gestor e criando automaticamente a Linha de Cuidado no Dynamics 365 (Viver Bem CRM).

A execução é feita via script (e não via aplicação OutSystems) porque o processo sempre foi conduzido de forma controlada, lote a lote, com validação manual a cada rodada. A decisão é não investir em desenvolvimento OutSystems enquanto o fluxo não estiver totalmente validado — o script Python é a forma mais rápida de iterar, ajustar filtros e corrigir o payload sem custo de deploy. A spec de migração para ODC (docs/SPEC_OUTSYSTEMS_ODC.md) só será executada depois que o processo estiver estável em produção.

⚠️ Ambiente atual CRM apontando para unimedpoa.crm2.dynamics.com (PRD). Linhas comentadas com HML (unimedpoa-hml…) e DEV (unimedpoa-dev…) ficam como referência no topo dos scripts 03_enviar_crm_linha_cuidado.py e 04_consulta_whatsapps_crm.py. Para alternar, descomentar a linha desejada e comentar a de PRD.
📁 Organização dos scripts Todos os arquivos .py ficam em scripts/ com prefixo numérico indicando a ordem de execução (01_ → 05_). Há um módulo compartilhado scripts/_vidas_api.py com as credenciais Sensedia e a função obter_token() reutilizada pelos scripts 01 e 05. Os arquivos de saída (CSVs, XLSXs, JSONs, log, cache) ficam na raiz do projeto.
7.1 Objetivo e Filtros

O script implementa o seguinte fluxo:

  1. Autenticar na API da Unimed POA (Sensedia, OAuth2 client_credentials).
  2. Buscar pacientes na base de Vidas (paginado, 25 em 25).
  3. Filtrar localmente: Alta Complexidade + sem gestor + sem cooperado + status "Não Navegado".
  4. Ordenar por hs_escore_priorizacao desc, manter os 100 primeiros.
  5. Para cada um: autenticar no Azure AD, buscar contactid por CPF no Dynamics, criar Linha de Cuidado.
  6. Logar cada passo (sucesso / erro / dry-run) em CSV.

Filtros aplicados localmente:

CampoRegra
complexidade= "Vidas Alta Complexidade"
gestorIS NULL
cooperado≠ "1"
ultimo_status_linha_cuidado_primaria= "Não Navegado"

Integrações utilizadas:

  • Sensedia (OAuth2) — POST /oauth2/v1/access-token
  • API Vidas — GET /analyticsBackoffice/v1/vidas (paginação 25 em 25)
  • Azure AD — POST /{tenant}/oauth2/token
  • Dynamics 365 — GET /contacts?$filter=uni_cpf eq '…' → POST /uni_linhadecuidados

Payload criado no CRM:

{
  "uni_estagiodajornada": 122880000,
  "uni_elegivelaoprogramacronico": true,
  "uni_name": "<nome do paciente>",
  "uni_Contact@odata.bind": "/contacts(<contactid>)"
}
7.2 Como Rodar

Pré-requisitos:

  • Python 3 (python --version) — sem dependências externas (só stdlib).
  • Opcional para XLSX formatado: pip install openpyxl.
  • Credenciais Sensedia + Azure AD (tokens obtidos em runtime via client_credentials).

1) Buscar pacientes na base de Vidas

# Pagina API de Vidas e filtra os 100 prioritários
python scripts/01_consulta_alta_complexidade.py

Pagina a API, aplica filtros locais, para nos 100 primeiros e ordena por hs_escore_priorizacao desc. Duração ≈ 2–3 min.

Saídas:

  • alta_complexidade_sem_gestor_nao_navegado.csv — lista final ordenada (entrada da etapa 3).
  • respostas_alta_complexidade/pagina_*.json — JSON bruto de cada página.

2) (Opcional) Gerar XLSX formatado

# Lê os JSONs brutos e gera planilha formatada
python scripts/02_gerar_xlsx_pacientes.py

Gera alta_complexidade_sem_gestor_nao_navegado.xlsx com cabeçalho azul, zebra, bordas, freeze pane e auto-filter.

3) Enviar ao CRM Dynamics

Modo dry-run (default — NÃO envia POST):

python scripts/03_enviar_crm_linha_cuidado.py                 # dry-run com todos os 100
python scripts/03_enviar_crm_linha_cuidado.py --limite 5      # dry-run só com 5

Modo real (exige flag explícita --confirmar):

python scripts/03_enviar_crm_linha_cuidado.py --confirmar --limite 1   # 1º teste real
python scripts/03_enviar_crm_linha_cuidado.py --confirmar --limite 5   # cria 5
python scripts/03_enviar_crm_linha_cuidado.py --confirmar              # cria todos os 100

Por paciente, o script faz:

  1. checar_linha_existente — se já tem linha, loga JA_POSSUI_LINHA e pula (não duplica).
  2. buscar_contato — se não acha CPF em contacts, loga SEM_CONTATO e pula.
  3. criar_linha_cuidado — em dry-run só monta payload; em modo real cria e grava uni_linhadecuidadoid.

Ordem recomendada para o primeiro uso:

python scripts/01_consulta_alta_complexidade.py                       # 1. busca
python scripts/03_enviar_crm_linha_cuidado.py --limite 5               # 2. dry-run
# 3. inspecionar log_crm_linha_cuidado.csv
python scripts/03_enviar_crm_linha_cuidado.py --confirmar --limite 1   # 4. 1 envio real
# 5. validar no Dynamics o uni_linhadecuidadoid do log
python scripts/03_enviar_crm_linha_cuidado.py --confirmar              # 6. resto

Travas de segurança:

  • Default é dry-run. Sem --confirmar, nunca envia POST.
  • Com --confirmar, espera 5s antes de começar (Ctrl+C aborta).
  • Erro num CPF é logado e o lote continua.
  • Log em append + flush imediato — dá pra acompanhar em outra janela com tail -f.

7.5 Relatório Enriquecido — Cruzamento Retornos × Base Vidas

Script (scripts/05_enriquecer_whatsapps_com_vidas.py) que cruza o CSV de retornos do WhatsApp (saída do script 04) com a base de Vidas para entregar ao time de Care Management uma planilha pronta para ação, com nome do beneficiário, gestor, complexidade, risco e telefone de cada CPF que respondeu.

O que o script faz:

  1. Lê retorno_whatsapps_crm.csv (gerado pelo script 04).
  2. Por default filtra para dias úteis (seg-sex); use --todos-dias para incluir fds.
  3. Extrai o CPF do campo description (padrão + CPF Disparo: XXXXXXXXXXX).
  4. Para cada CPF único, consulta a API de Vidas e captura: nome, gestor, complexidade, hs_risco, hs_escore_priorizacao, telefone_1, cooperado, ultimo_status_linha_cuidado_primaria. Cacheado em cache_vidas_gestor.json (re-runs não rebatem na API).
  5. Categoriza cada resposta do paciente em uma das 6 classes (regra heurística por texto): Resposta real, Resposta curta, Cumprimento vazio, Adia, Negativo, Sistema.
  6. Gera retorno_whatsapps_crm_enriquecido.xlsx com 5 abas (formatado, com freeze pane, auto-filter, cores semânticas e zebra).

Estrutura do XLSX (5 abas):

AbaConteúdo
ResumoMétricas do período: total de mensagens, CPFs únicos, % por categoria, top templates, contagem por dia.
Acionáveis ⭐Lista priorizada (categoria ≠ "Resposta real") ordenada por Alta Complexidade → Risco → Escore. Linha vermelha = Alta Complexidade pendente. Amarela = sem gestor atribuído.
Por_PacienteUma linha por CPF. Agrega quantas mensagens recebeu, distribuição de categorias, última data e última resposta. Verde claro = paciente teve ao menos uma "Resposta real".
Por_GestorMétricas por gestor: total de mensagens, CPFs únicos, respostas reais, quantos "a acionar", % a acionar, alta complexidade.
MensagensDetalhe linha-por-linha (raw) para auditoria.
