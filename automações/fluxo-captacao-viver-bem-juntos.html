🐍 Script de captação — Alta Complexidade → CRM Viver Bem

Script Python que alimenta a fila de captação a partir da base de Vidas da Unimed POA, selecionando os 100 pacientes prioritários de Alta Complexidade sem gestor e criando automaticamente a Linha de Cuidado no Dynamics 365 (Viver Bem CRM).

Objetivo

  1. Autenticar na API da Unimed POA (Sensedia, OAuth2 client_credentials).
  2. Buscar pacientes na base de Vidas (paginado, 25 em 25).
  3. Filtrar localmente por: Alta Complexidade + sem gestor + sem cooperado + status "Não Navegado".
  4. Ordenar por hs_escore_priorizacao desc, manter os 100 primeiros.
  5. Para cada um: autenticar no Azure AD (CRM), buscar contactid por CPF no Dynamics, criar Linha de Cuidado.
  6. Logar cada passo (sucesso / erro / dry-run) em CSV de log.
⚠️ Ambiente atual: CRM apontando para unimedpoa-hml.crm2.dynamics.com (HML). Antes de virar produção, alterar a constante CRM_RESOURCE e validar com dry-run.

Filtros aplicados

CampoRegra
complexidade= "Vidas Alta Complexidade"
gestorIS NULL
cooperado≠ "1"
ultimo_status_linha_cuidado_primaria= "Não Navegado"

Integrações utilizadas

SistemaUso
Sensedia (OAuth2)POST /oauth2/v1/access-token — obtém token Bearer
API VidasGET /analyticsBackoffice/v1/vidas — paginação 25 em 25
Azure ADPOST /{tenant}/oauth2/token — obtém token do CRM
Dynamics 365GET /contacts?$filter=uni_cpf eq '…' → POST /uni_linhadecuidados

Payload criado no CRM (POST /uni_linhadecuidados)

{ "uni_estagiodajornada": 122880000, "uni_elegivelaoprogramacronico": true, "uni_name": "<nome do paciente>", "uni_Contact@odata.bind": "/contacts(<contactid>)" }

▶️ Como rodar o script

Pré-requisitos

  • Python 3 (python --version) — sem dependências externas (só stdlib).
  • Opcional para XLSX formatado: pip install openpyxl.
  • Credenciais Sensedia + Azure AD (tokens obtidos em runtime via client_credentials).

Fluxo completo

1
Buscar pacientes na base de Vidas
python consulta_alta_complexidade.py

Pagina a API de Vidas, aplica os filtros locais, para nos 100 primeiros que passam e ordena por hs_escore_priorizacao desc. Duração ≈ 2–3 min.

Saídas:

  • alta_complexidade_sem_gestor_nao_navegado.csv — lista final ordenada (entrada da etapa 3).
  • respostas_alta_complexidade/pagina_*.json — JSON bruto de cada página.
2
(Opcional) Gerar XLSX formatado
python _gerar_xlsx.py

Lê os JSONs, refaz o filtro e gera alta_complexidade_sem_gestor_nao_navegado.xlsx com cabeçalho azul, zebra, bordas, freeze pane e auto-filter.

3
Enviar ao CRM Dynamics

Modo dry-run (default — NÃO envia POST):

python enviar_crm_linha_cuidado.py # dry-run com todos os 100 python enviar_crm_linha_cuidado.py --limite 5 # dry-run só com 5

Modo real (exige flag explícita --confirmar):

python enviar_crm_linha_cuidado.py --confirmar --limite 1 # 1º teste real python enviar_crm_linha_cuidado.py --confirmar --limite 5 # cria 5 python enviar_crm_linha_cuidado.py --confirmar # cria todos os 100

Por paciente, o script faz:

  1. checar_linha_existente — se já tem linha, loga JA_POSSUI_LINHA e pula (não duplica).
  2. buscar_contato — se não acha CPF em contacts, loga SEM_CONTATO e pula.
  3. criar_linha_cuidado — em dry-run só monta payload; em modo real cria e grava uni_linhadecuidadoid.

Ordem recomendada para o primeiro uso

python consulta_alta_complexidade.py # 1. busca python enviar_crm_linha_cuidado.py --limite 5 # 2. dry-run # 3. inspecionar log_crm_linha_cuidado.csv python enviar_crm_linha_cuidado.py --confirmar --limite 1 # 4. 1 envio real # 5. validar no Dynamics o uni_linhadecuidadoid do log python enviar_crm_linha_cuidado.py --confirmar # 6. resto

Status possíveis no log log_crm_linha_cuidado.csv

StatusSignificado
INFOMarcadores de início/fim de lote ou "processando"
OKEtapa intermediária deu certo (achou contato, sem linha prévia)
JA_POSSUI_LINHAPaciente já tem linha de cuidado — pulado
SEM_CONTATOCPF não encontrado em contacts do CRM
DRY_RUNPayload montado, mas POST não enviado
CRIADOPOST enviado e linha criada (modo --confirmar)
ERROFalha HTTP/parse — registrada e batch segue

Travas de segurança

  • Default é dry-run. Sem --confirmar, nunca envia POST.
  • Com --confirmar, espera 5s antes de começar (Ctrl+C aborta).
  • Erro num CPF é logado e o lote continua.
  • Log em append + flush imediato — dá pra acompanhar em outra janela com tail -f.
