Disparo de Mensagens via WhatsApp

Comunicação proativa com beneficiários — questionários e pesquisas pós-cirúrgicas
⚡ Dynamics + Blip ✅ Ativo
1

Visão Geral e Objetivo de Negócio

O objetivo desta integração é automatizar a comunicação entre a Unimed POA e seus beneficiários através do WhatsApp, utilizando a plataforma Blip como intermediária. A solução visa agilizar a coleta de informações assistenciais e o monitoramento de pacientes através do envio proativo de mensagens.

Templates Disponíveis
TemplateFinalidadeFlag de ativação
Desfecho Cirúrgico Pesquisa de satisfação enviada ao paciente após procedimento cirúrgico uni_desfecho_cirurgico: "true"
Universal (Questionários) Envio de pesquisas, formulários de saúde e questionários pendentes uni_questionariospendentes: "true"
2

Arquitetura e Fluxo de Dados

A integração segue um modelo de orquestração onde o CRM Dynamics atua como ponto central de comando:

  1. Origem — Viver Bem: o sistema busca os dados, realiza o tratamento necessário e dispara a requisição para o CRM.
  2. Processamento — Dynamics CRM: o CRM recebe o gatilho, valida o contato e prepara os parâmetros para a Blip.
  3. Execução — Blip: o CRM consome a API da Blip, que efetua a entrega final ao WhatsApp do cliente.

Especificações Técnicas (API)

3.1 Autenticação

Obtenção do token de acesso via Azure AD. Atentar-se à URL de ambiente (HML ou PRD).

CampoValor
Método POST
Endpoint https://login.windows.net/{{tenantid}}/oauth2/token
Resource HML https://unimedpoa-hml.crm2.dynamics.com
3.2 Identificação do Beneficiário

Antes de processar o envio, realiza-se a busca pelo CPF para garantir que o campo telephone1 (celular) esteja atualizado no CRM.

CampoValor
Método GET
Filtro {{resource}}/api/data/v9.2/contacts?$select=fullname,telephone1&$filter=uni_cpf eq 'CPF_VALOR'
Campo validado telephone1 — celular do beneficiário
3.3 Gatilho de Disparo — PATCH na Linha de Cuidado

O envio é realizado através de uma operação PATCH sobre um registro existente da entidade uni_linhadecuidados. A linha de cuidado precisa já existir para o beneficiário (foi criada anteriormente pelo fluxo de captação Viver Bem Juntos ou por outra automação).

CampoValor
Método PATCH
Endpoint {{resource}}/api/data/v9.2/uni_linhadecuidados({id})
Como obter o {id} GET /uni_linhadecuidados?$expand=uni_Contact($select=uni_cpf)&$filter=(uni_Contact/uni_cpf eq 'CPF')&$top=1
Headers obrigatórios Authorization: Bearer {{token}}, Content-Type: application/json, OData-MaxVersion: 4.0, OData-Version: 4.0, If-Match: *
⚠️ Não duplica registros Sempre PATCH (atualização). Se a linha de cuidado não existir para o CPF, crie primeiro via POST (ver projeto Automações — Alta Complexidade, seção 7) ou faça o tratamento de erro 404 antes de disparar o template.
3.3a Payload — Envio de Desfecho Cirúrgico

A flag uni_desfecho_cirurgico: "true" ativa o template de pesquisa de satisfação pós-cirúrgica na Blip. 

{
  "uni_variaveis_whatsapp": "[\"Douglas\", \"Cirurgia Mão\", \"Hosp. Clinicas\", \"12/03/2026\", \"453\"]",
  "uni_desfecho_cirurgico": "true",
  "uni_questionariospendentes": "false"
}

Ordem das variáveis em uni_variaveis_whatsapp:

PosiçãoConteúdoExemplo

3.3b Payload — Envio de Questionário Avulso

A flag uni_questionariospendentes: "true" ativa o template Universal na Blip, usado para envio de questionários e formulários de saúde sob demanda.

  "uni_questionariospendentes": "true",
  "uni_desfecho_cirurgico": "false"

3.3c Exemplo Completo (cURL)

Disparo de Desfecho Cirúrgico para uma linha de cuidado existente: 

curl -X PATCH \
  "https://unimedpoa-hml.crm2.dynamics.com/api/data/v9.2/uni_linhadecuidados(GUID-DA-LINHA)" \
  -H "Authorization: Bearer eyJ0eXAi..." \
  -H "Content-Type: application/json" \
  -H "OData-MaxVersion: 4.0" \
  -H "OData-Version: 4.0" \
  -H "If-Match: *" \
  -d '{
    "uni_variaveis_whatsapp": "[\"Douglas\", \"Cirurgia Mão\", \"Hosp. Clinicas\", \"12/03/2026\", \"453\"]",
    "uni_desfecho_cirurgico": "true",
    "uni_questionariospendentes": "false"
  }'

Regras de Disparo e Validação

RegraDescrição
Gatilho Imediato O envio para a Blip ocorre de forma imediata após o CRM receber os dados e a flag do template ser ativada.
Ordem das Variáveis O array uni_variaveis_whatsapp deve conter os valores na ordem exata configurada nos templates da Blip.
Validação do Celular O campo telephone1 deve estar preenchido no CRM antes do disparo; caso contrário, o envio não é processado.
Seleção do Template Apenas uma flag pode ser "true" por requisição — uni_questionariospendentes ou uni_desfecho_cirurgico.
ℹ️ Método POST vs PATCH Verificar sempre a existência da uni_linhadecuidados antes do disparo para evitar duplicidade de registros no CRM.

Pendências

⚠️ Gatilhos de envio ao CRM ainda não definidos A integração técnica (autenticação, payload, endpoint PATCH em uni_linhadecuidados) está mapeada e funcional, mas os gatilhos de negócio que disparam cada template ainda precisam ser definidos com o time clínico / operacional.

Perguntas em aberto a alinhar:

  • Desfecho Cirúrgico — em que momento disparar? (alta hospitalar, D+1, D+7?) Quem identifica o evento de alta? (HMAP, CRM, sistema do hospital?) Existe janela mínima/máxima após a cirurgia?
  • Questionário Avulso — qual a régua de envio? (sob demanda manual pelo gestor, agendado por período, automático ao mudar de linha de cuidado?) Como evitar disparo duplicado para o mesmo questionário?
  • Fonte dos dados das variáveis — quem alimenta uni_variaveis_whatsapp antes do PATCH? (workflow no CRM, integração externa, fila intermediária?)
  • Tratamento de exclusão — pacientes com opt-out (Viver Bem Juntos) devem ser bloqueados também aqui? Como cruzar?
  • Janelas de envio — respeitar horário comercial? Bloquear fins de semana / feriados?
  • Métricas de sucesso — o que medir? (taxa de resposta, NPS pós-cirúrgico, questionários respondidos vs. enviados?) Onde consultar?
📋 Próximos passos Reunião com time clínico + Care Management para mapear cada gatilho, frequência e critérios de elegibilidade. Documentar como seção adicional aqui assim que decidido.
